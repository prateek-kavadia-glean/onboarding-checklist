          Path reference

          1. Native connector

          • Best default for broad coverage, permissions, ranking, and support.
          • Check where the data actually lives before discussing custom work.
          • Best fit for large-team or department-wide workflows.

          2. Another system already collecting the data

          • Use when operational truth already lands in a standard system.
          • Great example: transcripts, summaries, or records written into Salesforce.
          • Often the fastest way to prove the workflow without bespoke work.

          3. API / MCP action

          • Fastest time to value.
          • Best for narrow lookups, actions, or real-time workflow bridges.
          • Lower quality than real indexed coverage; not a substitute for a broad connector.

          4. Partner-led build

          • Good early when the connector is essential and the customer is not technical enough to DIY.
          • Protects deal velocity and avoids overcommitting Glean in pre-sales.
          • Think of this as a dedicated workstream, not an add-on.

          5. Customer DIY on Glean APIs

          • Strong option late in cycle or with technically savvy customers.
          • SE helps with shape, permissions, IDs, URLs, and testing approach.
          • Customer owns build and maintenance.

          6. Glean-built custom connector

          • Last option, not first option.
          • Only after business scoping, technical alignment, and blocker status are clear.
          • Especially for file shares, this is a scoped project with real dependencies and calendar time.
          Stage guidance

          Stage 1 — Discovery

          • Validate pain, required capabilities, and whether the workflow matters broadly.
          • Do not dive into connector design.
          • Confirm where the data lives and what the customer is actually trying to prove.

          Stage 2 — Scoping / business value alignment

          • Map the workflow, quantify pain, and decide must-have vs nice-to-have.
          • This is the right stage to choose native vs mirrored source vs action vs partner vs DIY vs custom.
          • Bias toward repeatable, universal flows.

          Stage 4 — Security & IT alignment

          • By now you should have a clear proposed path.
          • Validate APIs, identity, permissions, volume, refresh, hosting, and constraints.
          • Use this call to confirm the path, not invent it.

          Stage 5 — Validation

          • Prove agreed success criteria.
          • If bespoke work is needed, it should already be framed as partner-led, DIY, or a formal custom workstream.
          • Do not improvise custom promises in validation.
